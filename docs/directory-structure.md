# Directory structure

The repository separates immutable configuration from persistent runtime data and the writable project workspace.

```text
docker-agentic-harness-sandbox/
├── Dockerfile                 # sandbox image and installed development tools
├── compose.yml                # sandbox, proxies, search, and storage services
├── start.sh                   # build and start helper
├── assets/                    # WeTTY favicon and visual alternatives
├── config/
│   ├── claude/                # Claude Code settings and global instructions
│   ├── opencode/              # provider, auth, agents, commands, and skills
│   ├── omp/                   # providers, model roles, MCP, and instructions
│   ├── searxng/               # search engine and limiter settings
│   └── litellm-config.yaml    # Claude alias to vLLM model mappings
├── data/
│   ├── claude/                # persisted Claude Code sessions and state
│   └── opencode/              # persisted OpenCode sessions and state
├── docs/                      # Docusaurus documentation pages
├── harness-proxy/             # experimental Rust Anthropic/OpenAI translator
├── ideas/                     # design notes and investigation drafts
├── patches/                   # build-time modifications to WeTTY
├── scripts/
│   ├── entrypoint.sh          # one-time container setup and supervision
│   ├── agent-session.sh       # per-browser session and tool selection
│   ├── agent-task.sh          # safe one-shot Claude Code wrapper
│   ├── claude-shim.js         # image-aware Claude-to-LiteLLM proxy
│   ├── upload-server.js       # upload service entry point
│   ├── upload-server/         # upload page assets and server implementation
│   └── reset-sandbox.sh       # destructive state reset helper
├── src/                       # Docusaurus landing page and styles
└── workspace/                 # host projects and uploaded images
```

## Configuration versus state

Files under `config/` define the intended sandbox behavior and belong in version control. Most are mounted read-only. Files under `data/` are generated by tools and persist across container replacement. Projects under `workspace/` are the only source trees the agents should edit by default.

The exception is OMP's main configuration and settings mounts, which are writable because OMP records its setup version. Treat unexpected changes there as generated state that should be reviewed before committing.

## Build-only files

The scripts under `patches/` modify the installed WeTTY package while the image is built. They are not copied into the final runtime filesystem. Alternative favicons under `assets/favicon-alternates/` are reference material; only `assets/favicon.ico` replaces WeTTY's active favicon.
